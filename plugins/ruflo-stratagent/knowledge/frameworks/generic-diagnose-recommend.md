# Generic diagnose-and-recommend (fallback)

## When this applies
The case doesn't cleanly match a named archetype, or genuinely combines
several. Use this to build a first-principles tree rather than forcing a
mismatched template.

## Default tree: the value driver tree
```
Value/Profit = Revenue - Cost - Capital cost
Revenue      = Price x Volume (x Mix)
Cost         = Fixed + Variable
Capital      = Working capital + Capex, weighed against required return
```

## How to build the tree for an unfamiliar case
1. **Restate the real question as a decision**, not a topic ("should we do
   X" or "why is Y happening and what do we do about it") — per
   case-classifier's output.
2. **Branch the value driver tree only as far as the question requires.**
   If the case is about cost, you may never need to touch the revenue side
   in depth, and vice versa.
3. **Check for a better-fitting named framework one level down.** A case
   that looks generic at first often turns out to be a profitability,
   pricing, or cost-reduction case once one layer of the tree is built —
   re-route to that cheat sheet if so.
4. **Keep it MECE**: each branch should be a distinct, non-overlapping
   driver of the top-line question.

## Common traps
- Forcing a famous framework (Five Forces, 7S, BCG matrix) onto a case it
  doesn't fit, producing analysis that's complete-looking but not actually
  responsive to the question asked.
- Building a tree so broad it can't be executed by the specialists
  available (financial/market/operations) in reasonable scope.
