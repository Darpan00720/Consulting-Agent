# M1.9 Completion Report

# Scope

M1.9 added **replay and recovery** on the frozen v1.0 architecture: a new
`packages/replay` sibling that rebuilds an append-capable `Engagement` from an
event log, and recovers a persisted engagement whose snapshot has fallen behind
the projection version. Replay is **pure orchestration over frozen seams** — it
composes `verify_log`/`verify_pair` (M1.7.4), `project` (M1.5/M1.7.2),
`AppendPipeline`/`make_committed` (M1.7.3), and reads decoded artifacts produced
by `EngagementStore` (M1.8) — and re-implements none of them. No ADR was
modified; `packages/state` and `packages/persistence` are **zero-diff** across
the whole milestone. Delivered across seven approval-gated phases under
`docs/implementation/M1.9-Design.md` (APPROVED).

# Replay architecture

Replay is a thin orchestration layer with the same posture `persistence` took
over its lower layers: **compose frozen seams, own no substantive logic.**

- **`ReplayEngine`** — a frozen, field-less, stateless value. `replay(log)`
  rebuilds from a bare log; `recover(log, snapshot)` rebuilds from a persisted
  pair. Both terminate in `AppendPipeline(state, log=log,
  append_supported=True)`, which invokes `make_committed` internally (P23) — so
  the engine never constructs `Committed` directly.
- **Canonical replay pipeline:** `verify_log → project → verify_pair →
  AppendPipeline → Engagement`. Because `project(log)` carries the current
  `PROJECTION_VERSION`, a verified log always passes `verify_pair`; replay of a
  verified log cannot fail at the fold/pair step.
- **Recovery:** operates on a **decoded** `(log, snapshot)` pair (not a store
  handle — `EngagementStore.load` rejects a stale snapshot itself). A valid pair
  rebuilds from the snapshot; a `PROJECTION_STALE` pair is discarded and the log
  re-projected once. All non-recoverable errors propagate unchanged.
- **Errors:** `ReplayError` is an additive orchestration-only base; integrity
  defects surface as the frozen `ReplayIntegrityError` (re-exported by
  identity). No new `ReplayErrorCode`.

Full reference: `docs/api/Replay.md`; layer view: `docs/architecture/replay-layer.md`.

# Benchmark summary

Single cold run, regression references only (never targets, no latency
assertions), same methodology as M1.7.7/M1.8-S5. Recorded in
`docs/performance/baselines.md`. At engagement scale (≤ ~1,000 events) all
operations are single-digit ms; at 10,000 events: `replay` ~126 ms (fold-bound),
`recover` on a **valid** pair ~5 ms (no re-projection), `recover` on a **stale**
pair ~132 ms (pays one re-projection ≈ `replay`), `verify_log`/`verify_pair`
~1.7–1.8 ms, `project` ~127 ms. Recovery adds no asymptotic cost over the seam
it wraps.

# Replay API surface (frozen)

Six public names (`tests/replay/test_api_freeze.py`): `ReplayEngine`,
`ReplayContract`, `replay`, `recover`, `ReplayError`, `ReplayIntegrityError`.
`ReplayEngine` public methods: `{replay, recover}`. `replay`/`recover` are also
module-level convenience functions. `ReplayError` extends `StratAgentError`;
`ReplayIntegrityError` is the frozen `state.append` object re-exported by
identity.

# Remaining technical debt

- **`verify_log` runs 2–3× per call.** The mandated order re-verifies the log
  inside each `verify_pair` (and again on the recovery re-verify). O(n) and
  harmless at scale; collapsing it would touch frozen `verify_pair` (out of
  scope).
- **RP-017 wording unconfirmed** (derived — see note above).
- **100k scale is an open decision.** Benchmarks cap at the approved 10,000;
  `replay`/`recover(stale)` are `project`-bound (~1.3 s extrapolated at 100k).
  Whether 100k is a supported scale or an explicit non-goal is a future call.
- **`verify_*`/`project` now have two recorded baselines** (M1.7.7 and this
  milestone); this report is the M1.9-context set.

# Readiness for the next milestone

The state lifecycle is now complete end to end: **create → append → validate →
persist (M1.8) → load → replay/recover (M1.9)**, all on the frozen v1.0
architecture. Log-backed engagements are append-capable again (the
`append_supported=False` adoption limitation is lifted via replay). Persistence
and replay compose cleanly: a caller loads a current engagement via
`EngagementStore.load`, or — when a snapshot has aged — decodes the artifacts and
calls `recover`, then persists the upgrade with `EngagementStore.save`. No
blockers for the next milestone.
