# M1.7 Completion Report

# Scope

M1.7 made the Engagement State **writable, safely**, on the frozen v1.0
architecture. It added: detached-snapshot reads, the guarded atomic append
pipeline and its public facade event API, fold-derived versioning, replay
integrity, the remaining gate-entry validation rules, the read/write ownership
matrix as data, and the official performance baselines — delivered across eight
approval-gated sub-milestones under the contract `docs/implementation/M1.7-Design.md`
(decisions D1–D6). No ADR was modified; the frozen v1.0 architecture is intact.

# Delivered Milestones

### M1.7.1 — Snapshot semantics (D1)
- **Objective:** stop `get_state()` leaking a mutable alias of internal state.
- **Implementation:** `get_state()` returns a deep detached copy; `from_state()`
  deep-copies on ingest. `facade.py` only.
- **Commits:** `a4612c8`, `79b3723`.
- **Tests:** snapshot-isolation suite (recursive identity, mutable-collection,
  top-level identity). **Evidence:** discharges checkpoint GO-condition TD-005.

### M1.7.2 — Fold-derived `state_version` (D4)
- **Objective:** make `state_version` a projection-derived fact, not a
  caller-set field.
- **Implementation:** `apply()` stamps `metadata.state_version = event.seq`;
  `PROJECTION_VERSION` 1→2 with a version-history entry + migration note.
- **Commits:** `d8ae1b6`, `4d149e0`.
- **Tests:** derivation/tamper-resistance/reprojection-identity. **Evidence:**
  first controlled exercise of the projection-versioning policy.

### M1.7.3 — Append pipeline + facade event API (D2/D3/D5/D6)
- **Objective:** the one, guarded, atomic mutation path.
- **Implementation:** `append/` package — `errors`, `result`, `sequencing`,
  `versioning`, `guard`, `commit`, `pipeline` — composing S3 admission/conflict
  decisions, S2 arithmetic, M1.7.2 projection, and M1.6 validation under the
  fixed order **Decision → Allocation → Projection → Validation → Commit**;
  `make_committed` the single `Committed` construction path. Facade gained
  `append_event`/`append_events`/`current_version`/`current_sequence` and
  `validate()→ValidationReport`; read-only adoptions raise `AppendUnsupportedError`.
  Delivered as slices **S1–S6** (S7/S8 dispositioned — below).
- **Commits:** `6e096dd`, `960a9e8`, `9dadd89`, `ad0bdd3`, `e7e4d6d`, `c874ad3`,
  `cc30156`, `0c7a32e` (+ `e67fc3c` decomposition).
- **Tests:** invariant suites S1-contract, A1–A8, V1–V7/C1–C3, G1–G16, P1–P24,
  F1–F10, plus the S6 API-freeze. **Evidence:** public API frozen at 10 methods.

# Readiness for M1.8

M1.8 (Persistence) may now assume:
- **A single, verified construction path** for committed state — `make_committed`
  (P21/P23); persistence restores through it, never by hand.
- **A ready at-rest gate** — `verify_log`/`verify_pair`: the canonical persisted
  artifact is exactly the `(log, project(log))` pair the checker validates.
- **A complete commit payload** — `CandidateCommit.events` is precisely what to
  persist; no reconstruction from `log[-k:]`.
- **The atomic-commit invariant to preserve** — commit is one reference swap;
  M1.8's durability ordering (write → fsync → advance) must keep it atomic.
- **Two scheduled inheritances:** event schema upcasting (TD-007) and the
  read-only-adoption restriction, which persisted logs + M1.9 replay remove.
- **Baselines to regress against** — `docs/performance/baselines.md`.

M1.7 is complete. **M1.8 is not started** (out of scope here).
