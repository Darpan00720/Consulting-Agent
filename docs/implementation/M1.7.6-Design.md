# M1.7.6 Design — Read/Write Ownership Matrix

# Purpose

Give the repository a single, machine-readable, completeness-tested source of
ownership truth — **data only, no executable enforcement** — so that (a) the
architectural boundaries built across M1.1–M1.7.5 can never drift silently,
and (b) M6's Agent Manager has a ready, tested dataset to enforce when roles
become real (the recorded TD-003 plan [Verified]).

# Repository Evidence

What the evidence pass found, by source:

- **ADR-002 §Agent Read/Write Matrix** [Verified — read in full]: a *complete*
  agent-vs-state-section permission model. Read = all agents (single source of
  truth; tenant-scoping only). Write/create = **owner-exclusive**. Update =
  owner (+ Manager for lifecycle). Approve/Reject = gatekeepers only (Reviewer,
  Challenger, Human/Manager). The event log: "append-only by all, mutable by
  none." 25 section rows, each with all five columns populated.
- **ADR-002 §Event Model** [Verified]: every catalogued event row carries a
  "By" (writing role) and "Effect" (projected section) column — a per-event
  ownership source the implementation has never transcribed.
- **ADR-005 §2–§3** [Verified]: 16 agent specifications with responsibilities;
  the role names used by ADR-002's matrix resolve to these specifications.
  No role registry exists in code [Verified: TD-003 deferral, `actor` is a
  free string].
- **Code-level component ownership** [Verified — but *prose and tests*, not
  data]: the S4 ownership matrix lives in `pipeline.py`/`commit.py`
  docstrings; single-writer facts are pinned by invariant tests (P9/P17/P19/
  P21/P23, F1–F3, G1); M1.7.4's integrity contract states the at-rest/write
  split; M1.7.5 §5.1 owns the rule table; the disposition table maps ADR rules
  to mechanisms. **None of this is queryable data.**
- **Knowledge layer** [Verified]: `grep` shows zero imports between
  `packages/state` and Graphify/Ruflo tooling; the vault is repository data
  indexed externally; ADR-003 gives the Knowledge Agent read access via
  retrieval interfaces (M3).
- **Traceability generator** [Verified]: `scripts/generate_traceability.py` is
  the only writer of the two traceability documents (`make traceability`).

# Problem

Ownership truth exists in **four disconnected forms** — ADR tables (prose),
module docstrings (prose), invariant tests (behavioral pins), and the
disposition table (partial data). There is no single dataset that a test — or
M6's enforcement — can consume. The specific gaps ("Missing Ownership" below)
are exactly what M1-Decomposition scoped for M1.7.6: "owner-exclusive section
writes (R/W matrix)" as **data** [Verified], per the approved D6 decision
("data now, enforcement at M6") [Verified].

# Missing Ownership

1. **Component-level ownership as data** — the 16-row matrix below exists
   nowhere as a queryable artifact [Verified: absence].
2. **The ADR-002 agent-section matrix as machine-readable data** — promised by
   D6, never transcribed [Verified].
3. **The event → section → writer transcription** (49 events) — same
   [Verified].
4. **Role enforcement** — deliberately absent until M6 [Verified: TD-003].
   Not a gap this milestone fills; every row below names M6's enforcement
   owner instead.

# Design Decisions (for approval)

- **DD-1 — Home:** a new internal data module `packages/state/ownership.py`
  containing three frozen datasets: `COMPONENT_OWNERSHIP` (the matrix below),
  `SECTION_OWNERSHIP` (ADR-002 agent matrix, transcribed verbatim), and
  `EVENT_OWNERSHIP` (event-type → affected section(s) → writing role(s), from
  the ADR-002 event table). Data only: plain frozen dataclasses/mappings; no
  function makes a decision. Not exported publicly (`state.__all__`
  untouched; S6 freeze passes unmodified).
- **DD-2 — Emission:** the traceability generator gains an ownership section
  (markdown + JSON) so the human-readable artifact and the data can never
  diverge (same pattern as TD-004 dispositions).
- **DD-3 — Completeness tests, not enforcement:** tests assert the shape of
  the data (below), not runtime behavior. The one behavioral fact tests may
  pin is what existing invariant tests already pin — referenced, not
  duplicated.
- **DD-4 — Role vocabulary:** `SECTION_OWNERSHIP`/`EVENT_OWNERSHIP` use the
  ADR-005 role names as string constants (a `Role` StrEnum), which becomes the
  seed of M6's role registry. [Inference: ADR-005 names are stable enough to
  freeze; flagged.]

Completeness properties the implementation's tests will pin:
- exactly **one** writer per writable resource (committed snapshot →
  StateUpdater; candidate → pipeline; stamped copies → allocator; report →
  runner; traceability artifacts → generator; vault graph → `vault-index`);
- **no overlapping write ownership** (asserted as pairwise-disjoint writable
  resources across rows);
- shared read is the default (ADR-002 read-all), stated per row;
- every row carries evidence and a future enforcement owner (tests today;
  M6 Agent Manager for role rows);
- `SECTION_OWNERSHIP` covers **every `EngagementState` section field** (drift
  test against `model_fields`); `EVENT_OWNERSHIP` covers **all 49 event
  types** (drift test against `EventType`).

# Definition of Done (implementation phase, after approval)

- **Scope:** `packages/state/ownership.py` (Role enum + three frozen
  datasets); generator emission (md + JSON `ownership` key); tests. No
  enforcement, no frozen-module changes, no public-API change.
- **Files:** new `packages/state/ownership.py`,
  `tests/state/test_ownership.py`; modified
  `scripts/generate_traceability.py`, regenerated traceability docs,
  `CHANGELOG.md`, this doc's status.
- **Tests:** dataset-shape completeness (the five properties above); ADR
  fidelity spot-checks (e.g., Reviewer Notes writer == Reviewer; event-log row
  == append-only-all/mutable-none); generator freshness extended.
- **Acceptance criteria:** all datasets complete and drift-tested; six-step
  gate green; 100% coverage on the new module; frozen files zero-diff.
- **Risks:** *Low* — transcription errors from ADR tables (mitigated by
  spot-check tests quoting the ADR verbatim); *Low* — role-name drift before
  M6 (the Role enum is additive-frozen like every namespace here).
- **Architectural concerns:** none — this milestone writes down what already
  exists and tests that it stays written down.
