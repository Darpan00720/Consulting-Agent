# M1.7.7 Design — Performance Baselines

# Purpose

Record **official, reproducible baselines** for the operations M1.7 introduced,
and close the one performance debt (TD-010). Baselines are **regression
references, not targets** — the standing policy since M1.5: no optimization
without a measurement that justifies it [Verified: projection-baseline.md,
M1.7-Design §7]. This milestone measures and documents; it changes no behavior.

# Evidence

  (10/100/1k/10k events) and `tests/perf/test_validation_bench.py`

- **Deferred benchmark work** [Verified]: M1.7.3-Decomposition **S8**

# Benchmark Methodology

- **Reuse the established methodology** [Verified precedent]: `pytest-benchmark`,
  `benchmark.pedantic(fn, rounds=1, iterations=1)` — one measured cold run per
  scale; scales `10 / 100 / 1,000 / 10,000`. This keeps the suite fast and the
  numbers directly comparable to the projection/validation baselines.
- **Honest caveats carried forward** [Verified precedent]: single-run ⇒ run-to-
  run variance (±30–50% at small N); machine-specific (Apple M3); baselines to
  catch gross regressions, **not** statistically rigorous micro-benchmarks.
- **Fixtures:** the append/`get_state` state of size N is built by projecting a
  valid N-event log (reusing the M1.7.4/M1.5 helpers), so the benchmarked state
  is realistic and the fixture code is not duplicated.
- **Not gates:** benchmarks assert only that the operation *ran* (e.g. an append
  succeeded), never a latency threshold — identical to the existing perf tests.

# Metrics

Per operation × scale: **mean** wall-clock (the pytest-benchmark headline) and a
derived **per-unit** figure (per-object for state-sized ops, per-event for
log-sized ops), matching the projection-baseline.md presentation. Environment
block (machine, OS, Python, pytest-benchmark version, date) captured once,
freshly, at measurement time — never hand-copied.

# Risks

- **Low — variance/machine-specificity**: inherent to single-run; mitigated by
  the documented caveats and the "regression reference, not target" framing.
- **Low — suite runtime**: four new parametrized benchmarks (~16 measured runs)
  add well under a second at these scales [Inference from existing bench
  timings: 10k-object validation ≈ 4.4 ms, 10k-event projection ≈ 33 ms].
- **None architectural**: no source, no frozen module, no public API, no ADR
  touched — measurement and documentation only.

  the standard caveats; cross-reference to projection-baseline.md.
- **Acceptance criteria:** benchmarks green in the full gate; numbers recorded
  in docs/performance; TD-010 closed; frozen files zero-diff; 100% coverage not
  applicable to benchmark files (they are perf tests, excluded from the coverage
  target as the existing perf tests are) — the six-step gate otherwise green.
- **Risks / architectural concerns:** none beyond the Low items above.
