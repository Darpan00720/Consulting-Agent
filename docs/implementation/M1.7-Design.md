# M1.7 Design — Concurrency, Versioning & Corrections

This document resolves every GO condition from the Architecture Checkpoint and
removes the architectural ambiguities identified there **before** any M1.7 code is
written. On approval it is the implementation contract: implementation may not
deviate from a Decision below without returning here first.

Scope boundary (unchanged from M1-Decomposition): **in** — monotonic `seq`,
`state_version`, optimistic concurrency, corrections only via events, the write
seam of the R/W matrix; **out** — distributed/multi-process coordination,
persistence (M1.8), replay/snapshots (M1.9).

---

# 1. Objectives

M1.7 makes the Engagement State **writable, safely**:

1. Introduce the **event-append API** — the first and only mutation path. State
   changes become facts in a log, never edits to an object.
2. Establish **ordering**: every admitted event gets a contiguous, monotonic `seq`;
   the log's seq order is the single source of truth for "what happened when".
3. Establish **versioning**: `state_version` becomes a live, log-derived property
   of the state, enabling `state_version == max(seq)` (ADR-002 invariant).
4. Establish **optimistic concurrency**: stale writers — writers acting on an
   outdated view — are detected and rejected, never merged.
5. Enforce **corrections only via events**: there is no API that edits or deletes;
   compensating events are the only way to change course.

**Why concurrency comes only now.** Concurrency is not a feature; it is a set of
guarantees *about* other things — and every one of those things had to exist first:

- **Engagement State (M1.1–M1.3)** defines *what* is being versioned. A version
  number over an undefined shape guarantees nothing.
- **Events (M1.4)** define the *write medium*. Optimistic concurrency's unit of
  conflict is the append; without an immutable, self-contained event there is
  nothing to sequence, and `seq`/`actor`/`occurred_at` had to exist on the
  envelope before an allocator could have anything to stamp.
- **Projection (M1.5)** defines *what a write means*. Conflict handling is only
  sound if applying the same events in the same order always yields the same
  state — determinism is the property that makes "reject and retry from a fresh
  read" a correct recovery, not a gamble.
- **Validation (M1.6)** defines *what an acceptable state is*. The append pipeline
  needs a judge: an append that would produce an invalid state must be refused,
  and that judgment had to exist as a pure, separate function before the pipeline
  could call it.

Introducing concurrency earlier would have meant redesigning the append pipeline
after each of these landed. Sequenced this way, M1.7 composes four finished, tested
parts and adds exactly one new idea: the guarded, atomic append.

# 2. Architectural Decisions

Six decisions. D1 and D2 discharge the two binding GO conditions; D3–D6 remove the
remaining ambiguities the checkpoint identified.

## D1 — Mutable State (GO condition 1: live-state aliasing, TD-005)

**Problem.** `Engagement.get_state()` returns the live internal `EngagementState`;
models are not frozen and list sections are mutable. A reader can mutate the very
object the writer versions, making any optimistic-concurrency guarantee unsound
(checkpoint §5/§13, classified **High**).

**Options considered.**

| Option | Assessment |
|---|---|
| **(a) Defensive copy on read (snapshot semantics)** | Every `get_state()` returns a deep, detached copy. Internal state is reachable only by the append pipeline. Simple, complete, no model changes. Cost: O(state) per read. |
| (b) Freeze the models (`frozen=True` everywhere) | **False confidence:** Pydantic `frozen` blocks attribute assignment but does **not** protect list contents — `state.evidence.append(...)` would still mutate shared state. Real protection requires migrating every `list[...]` field to `tuple[...]` across ~25 section models: invasive, schema-churning, and ergonomically hostile, for the same guarantee (a) gives in one place. |
| (c) Read-only wrapper/proxy | Unidiomatic in Python; defeats mypy strictness (proxies erase types) and remains bypassable via `__wrapped__`-style access. Guarantee by obscurity. |
| (d) Copy-on-write | Right answer at a scale we do not have. Requires custom collection types and mutation tracking; the entire state copies in microseconds today (§7). Complexity without a measured need — contrary to the project's baseline-first policy. |

**Decision: (a) — snapshot on read, at both boundaries.**

- `get_state()` returns a **detached snapshot** (deep copy). Mutating a snapshot
  never affects the engagement. Two calls return equal but distinct objects.
- `from_state(state)` **deep-copies on ingest** — the caller's retained reference
  is not an alias of the engagement's internal state.
- `to_json()` serializes the internal state directly (serialization cannot alias).
- The internal state is mutated by exactly one component: the State Updater (§4),
  and only at the commit point of an append.

**Rationale.** This is the event-sourcing-correct reading of the M1.3 approval
("do not expose live EngagementState … use `get_state()`/snapshot()"): in an
event-sourced system a materialized state is *always* a derived view; handing out
views as immutable-by-detachment snapshots matches the model exactly. It makes
optimistic concurrency sound by construction — a writer's `expected_version` refers
to a snapshot that *cannot* have drifted underneath it — and it needs no changes to
25 models, no schema drift, and no new idioms.

**Consequences.** Reads acquire an O(state) copy cost (measured in §7 benchmarks;
microseconds at engagement scale). Callers that previously relied on aliasing
(none exist — no external consumers yet, checkpoint §9) would break; this is a
pre-release behavioral tightening of an approved intent, documented in
`docs/api/EngagementState.md`. Facade method signatures are unchanged.

## D2 — Sequence Numbers (GO condition 2: seq/ordering contract, TD-006)

**Problem.** `EventMetadata.seq` defaults to `0` with no assignment authority, no
monotonicity enforcement, and no documented ordering contract; projection folds in
caller-supplied order and never reads `seq` (checkpoint §6/§7).

**Decision — the seq contract:**

- **Who assigns:** the **Sequence Allocator** inside the append pipeline of the
  owning `Engagement` instance. Nothing else ever assigns `seq`. Callers **must
  not** pre-assign: an incoming event with `seq != 0` is rejected at admission
  (`0` is hereby defined as the *unassigned sentinel* — it is only legal on an
  event that has not yet been admitted, resolving checkpoint §6 weakness 1).
- **When:** at append time, *after* the Concurrency Guard passes and *before* the
  event enters the candidate log. Because events are frozen, the allocator stamps
  a **copy** of the submitted event (`seq` filled in); the caller's instance is
  never touched. The stored, stamped copy is the event of record.
- **Uniqueness:** `seq` is unique per engagement, by construction — a single
  allocator per engagement, allocation only inside the serialized append pipeline.
- **Ordering:** `seq` starts at **1** and is **strictly monotonic and contiguous**
  (`1, 2, 3, …` — no gaps, no duplicates). Gaps are impossible because allocation
  and commit are one atomic step (D6): a failed append never consumes a number.
- **Replay:** `project()` remains exactly as approved in M1.5 — a pure fold in
  iteration order that does not read `seq` (no projection redesign). The ordering
  contract lives **at the boundary**: any component that feeds a log to
  `project()` (the M1.7 rebuild path, M1.9 replay, M1.8 load) must first pass it
  through an **integrity check** — `events[i].metadata.seq == i + 1` for all i —
  and refuse to fold on violation (`ReplayIntegrityError`). This precondition is
  documented on `project()` and in `docs/api/Events.md`.

**Single source of truth for ordering: the committed event log's `seq`, as
allocated by the append pipeline.** Not `occurred_at` (business time can be out of
order — a backdated fact is legal), not `recorded_at`, not list position in any
transient collection.

**Rationale.** One allocator per aggregate is the standard event-sourcing answer
and the only one that gives contiguity without coordination. Keeping projection
seq-blind preserves M1.5's purity and its version-1 semantics; enforcing order at
the boundary puts the check where the authority is.

**Consequences.** Existing tests that hand-build logs with default `seq=0` remain
valid for *projection* tests (projection is order-taking) but will not pass the
boundary integrity check — append/replay tests must build properly sequenced logs.
The contract text is the fix for TD-006.

## D3 — Optimistic Concurrency

**Problem.** ADR-002 mandates optimistic concurrency (stale-version reject +
retry) but leaves the algorithm undefined.

**Decision — the algorithm (no implementation):**

- **`expected_version`** is a **required** integer parameter on every append: the
  version of the state the writer read when it decided to write. There is no
  unconditional append in M1.7 (no "last writer wins", no `None` escape hatch);
  if commutative appends ever justify one, that is an additive API change to be
  approved then.
- **Conflict detection:** at admission, the Concurrency Guard compares
  `expected_version` to the committed current version (an O(1) integer compare).
  - `expected == current` → proceed.
  - `expected < current` → **stale writer**: another append committed after this
    writer's read. Rejected with `VersionConflictError(expected, actual)`. Never
    merged, never partially applied.
  - `expected > current` → also a conflict (the writer claims a version that has
    never existed — a corrupted or confused caller). Same rejection, same error.
- **Retry behavior:** the state layer does **not** retry. On conflict the caller
  must re-read (`get_state()` + `current_version()`), **re-derive its event(s)
  from the fresh state** (the old events may no longer make sense), and append
  again with the new expected version. Retry *policy* — how many times, backoff,
  escalation to human — belongs to the orchestrating layer (Engagement Manager,
  M6+), not to `state`. The error therefore carries both versions so any policy
  can be built on it.
- **Batch:** `append_events` takes **one** `expected_version`, checked once
  against the batch as a whole; the batch commits or fails atomically (D6).
- **Scope of the guarantee (honest statement):** M1.7 defends against **stale
  reads by interleaved logical writers** — multiple agents taking turns through
  one `Engagement` instance — which is the concurrency model of ADR-001 (the host
  executes agents via the Task tool; writes funnel through the manager).
  **OS-thread safety is out of scope**: one writer thread per engagement instance
  is a documented usage constraint, consistent with M1-Decomposition's exclusion
  of distributed/multi-process concerns.

**Rationale.** Compare-and-reject on a single integer is the simplest algorithm
that satisfies ADR-002's rule; every added sophistication (merge, auto-retry,
per-section versions) embeds policy the state layer has no business owning.
Per-section version vectors were considered and rejected: ADR-002's own answer to
section contention is owner-exclusive sections (disjoint writers), which makes a
single aggregate version sufficient and much simpler to reason about.

**Consequences.** Writers must tolerate rejection as a normal outcome (it is a
signal, not an error state). High contention degrades to retry loops — acceptable
at engagement scale (a handful of agents), and measurable via the §7 benchmarks.

## D4 — `state_version` derivation (and the first `PROJECTION_VERSION` bump)

**Problem.** `EngagementMetadata.state_version` exists but nothing maintains it;
ADR-002 requires `state_version == max(events.seq)`. Who sets it — the pipeline
(side-channel stamp after fold) or the fold itself?

**Decision:** the **fold itself**. `apply(state, event)`'s contract is extended:
after dispatching the event's reducer, it sets
`metadata.state_version = event.metadata.seq`. `state_version` thereby becomes a
pure, deterministic, log-derived property — replay automatically restores it, and
`project(log).metadata.state_version == log[-1].metadata.seq` by construction
(0 for the empty log, matching "no events").

Because this changes the fold's output for previously-valid logs, it triggers the
documented versioning policy: **`PROJECTION_VERSION` bumps 1 → 2** — the first
controlled exercise of `docs/architecture/projection-versioning.md`, with the
required CHANGELOG entry. Per-event reducers are untouched; the stamp happens once
in `apply`.

The ADR invariant `state_version == max(seq)` is then enforced at the **boundary**
(it needs the log *and* the state, so it cannot be a state-only validation rule):
the replay/rebuild integrity check of D2 verifies it alongside contiguity, and the
traceability disposition (D6) records it as boundary-enforced rather than
registry-enforced.

**Rejected alternative:** State-Updater stamping after the fold. It works, but it
makes `state_version` a side-channel fact that raw `project()` output lacks —
replay in M1.9 would silently produce version-less states unless every caller
remembered to stamp. Deriving it in the fold makes forgetting impossible.

**Consequences.** One projection-module change (in `apply`, not in reducers), one
version bump executed exactly per policy, projection tests updated to assert the
stamp, and the invariant triangle becomes testable:
`current_version() == get_state().metadata.state_version == len(log) == last seq`.

## D5 — Validation's role in the append pipeline

**Problem.** M1.6 validation exists but nothing calls it; the checkpoint noted
`Engagement.validate()` still performs Pydantic re-validation only. What blocks an
append?

**Decision:** every append validates the **candidate post-state** with
`state.validation.validate()`:

- Blocking violations (**ERROR/FATAL**, i.e. `report.valid == False`) → the append
  is rejected (`StateValidationError` carrying the report), nothing committed.
- **WARNING/INFO** violations do not block; they are returned to the caller in the
  `AppendResult` so the orchestrator can surface them.
- The rule is deliberately stated on the **post-state, not the delta**: an append
  is acceptable iff the state it produces is acceptable. This has an important
  corollary — if a state is already invalid (e.g., a legacy import), the only
  admissible events are **corrective** ones (whose post-state is valid), which is
  exactly ADR-002's "corrections only via events" made mechanical.
- `Engagement.validate()` is upgraded to run invariant validation and return the
  `ValidationReport` (surfacing M1.6 through the facade, as planned "in a later
  milestone" — this is that milestone). Its previous raise-on-schema-error
  behavior is subsumed: Pydantic validity is a precondition of constructing the
  state at all.

**Consequences.** Appends pay one validation pass (~71µs–4.4ms at measured scales,
§7). `validate()`'s return type changes from `None` to `ValidationReport` — a
pre-release, documented facade refinement (the protocol is updated in lockstep).

## D6 — Atomicity, admission, and scheduled-debt scope

**Atomic prepare-commit.** An append never mutates committed state in place. The
pipeline builds a **candidate**: stamped event(s) + the new state produced by
incremental `apply` on the committed state (projection is pure, so the committed
state is never touched by a failed attempt). Only after allocation, application,
and validation all succeed does the State Updater **commit** — a single atomic
replacement of (log, state, version) — the one mutation point in the system.
Failure at any earlier step discards the candidate; the engagement is
byte-identical to before the attempt, and no `seq` was consumed. **Partial append
is impossible by construction**, for single events and batches alike.

**Admission checks** (Concurrency Guard, before allocation): the event's
`engagement_id` matches the engagement; `seq == 0` (unassigned, D2); the
`event_id` does not already exist in the log (idempotency guard — an accidental
double-submit of the same event instance is rejected; a legitimate post-conflict
retry constructs fresh events from a fresh read, so it is unaffected).

**Owner-exclusive writes (R/W matrix) — data now, enforcement at M6.** ADR-002's
agent Read/Write matrix requires knowing *which role* is writing. Verified fact:
`EventMetadata.actor` is a free string and `EventSource` is a channel
(cli/api/agent/…), not a role — a role registry is ADR-005 material that does not
exist until agents do (M4–M6). Inventing a role system inside M1.7 would be
architecture creep. M1.7 therefore ships the matrix as **machine-readable data**
(event-type → owning section(s) → owning role, transcribed from ADR-002) with a
completeness test (every `EventType` mapped), and the Concurrency Guard gains the
*seam* where the role check will sit; enforcement activates at M6 when roles are
real. This honors the checkpoint's "keep it data, not behavior".

**Scheduled debt folded into M1.7** (from the checkpoint's non-blocking list):

| Item | Disposition in M1.7 |
|---|---|
| TD-002 gate-entry precondition rules | **In scope**: `LIFE-005` (Enter Planning), `LIFE-006` (Analysis-entry: non-empty owned tree + plan), `LIFE-007` (Reviewer-entry: leaves answered), `LIFE-008` (Challenger-entry: review approved) — new frozen ids, appended to the lifecycle registry, matrix regenerated. |
| TD-003 approval-actor rules | **Deferred to M6** with recorded rationale: requires the role registry (above); the data they need (`QualityGate.by`, populated from `actor` — projection.py:467) is already captured, so no information is lost by waiting. |
| TD-004 traceability dispositions | **In scope**: the generator gains a disposition table for non-registry ADR-002 rules (record-level / boundary-enforced / deferred-to-milestone). |
| TD-008 tooling targets 3.11 vs runtime 3.12 | **In scope** (chore commit). |
| TD-009 decomposition doc inconsistency | **In scope** (doc fix with the M1.7 commits). |
| TD-010 validation-benchmark baseline doc | **In scope** (extends docs/performance). |

# 3. Public API (contracts only — no implementation)

The facade grows by the **event API that M1.3 explicitly reserved** (facade
docstring: "state evolves only through the event API (a later sub-milestone)");
this is that pre-authorized extension, mirrored in `EngagementProtocol`.

**New result/error contracts**

- `AppendResult` (frozen model): `version: int` (committed version after the
  append), `first_seq: int`, `last_seq: int`, `appended: int` (event count),
  `warnings: list[Violation]` (non-blocking findings from D5).
- `AppendError(StratAgentError)` — base for append failures.
  - `VersionConflictError(AppendError)` — carries `expected: int`, `actual: int`.
  - `EventAdmissionError(AppendError)` — engagement mismatch, pre-assigned seq,
    or duplicate `event_id`; carries the offending event's id and the reason.
- `StateValidationError` (existing, M1.6) — raised when the candidate post-state
  is invalid; carries the `ValidationReport`.
- `ReplayIntegrityError(StratAgentError)` — boundary ordering-check failure (D2).

**`append_event(event: Event, *, expected_version: int) -> AppendResult`**
Appends one event. Contract: runs the full pipeline (§5); on success the event —
stamped with the next contiguous `seq` — is the newest entry in the log, the
committed state reflects it, and `result.version == expected_version + 1`. On any
failure, raises one of the errors above and **the engagement is unchanged**. The
caller's event instance is never mutated (stamping copies, D2).

**`append_events(events: Sequence[Event], *, expected_version: int) -> AppendResult`**
Atomic batch append. Contract: one guard check against `expected_version`;
contiguous seqs across the batch (`first_seq == expected_version + 1`,
`last_seq == expected_version + len(events)`); events applied in the given order;
**one** validation of the final post-state (intermediate states are not judged —
the batch is one logical write); all-or-nothing commit. Empty batch is an
`EventAdmissionError` (a no-op append is a caller bug, not a version bump).

**`current_version() -> int`**
The committed version: the `seq` of the last committed event; `0` for a fresh
engagement. Invariant: equals `get_state().metadata.state_version` (D4) and the
committed log length (D2). O(1); never raises.

**`current_sequence() -> int`**
The next `seq` the allocator will assign: always `current_version() + 1`. Provided
so callers never compute sequence arithmetic themselves. O(1); never raises.

**Changed contracts (documented refinements, D1/D5):**
`get_state()` — returns a detached snapshot. `from_state()` — deep-copies its
input. `validate()` — returns `ValidationReport` (was `None`).
`docs/api/EngagementState.md` and `Events.md` are updated in the same milestone;
the export-surface allowlist test grows by exactly the names above.

# 4. Internal Components (responsibilities only)

- **Version Manager** — owns the committed version number. Answers
  `current_version`/`current_sequence`; supplies the comparison value to the
  Concurrency Guard; advances only at commit. The single authority on "what
  version are we at".
- **Sequence Allocator** — turns admitted events into stamped events: assigns
  `expected_version + 1 … + k` contiguously, produces the frozen stamped copies.
  Allocates only inside a pipeline run; a discarded candidate returns nothing to
  roll back because numbers only become real at commit.
- **Append Pipeline** — the only orchestrator (mirror of the M1.6 runner
  principle: components stay independent; sequencing lives in exactly one place).
  Runs admission → guard → allocate → apply → validate → commit, owns the
  candidate, translates step failures into the §3 error contracts.
- **Concurrency Guard** — pure decision-maker: admission checks (engagement
  match, unassigned seq, unknown `event_id`) and the `expected_version` compare.
  Holds the future role-check seam (D6). Answers yes/no with a reason; never
  mutates anything.
- **State Updater** — the commit point. Atomically replaces the committed
  (log, state, version) triple with the candidate's. The only component that ever
  mutates the engagement's internals; everything upstream works on copies.

# 5. Append Workflow

```
Client → Facade → Concurrency Guard → Sequence Allocation
       → Append (candidate log) → Projection (incremental apply)
       → Validation → Commit → Updated State
```

1. **Client** reads a snapshot + `current_version()`, decides, constructs
   event(s) (`seq=0`, correct `engagement_id`, fresh `event_id`s), calls
   `append_event(s)(…, expected_version=v)`.
2. **Facade** hands the call to the Append Pipeline — no logic in the facade.
3. **Concurrency Guard** — admission (D6) then version compare (D3). Failure →
   `EventAdmissionError`/`VersionConflictError`; nothing allocated.
4. **Sequence Allocation** — stamped copies with `v+1 … v+k` (D2).
5. **Append (candidate)** — stamped events appended to a candidate log; the
   committed log is untouched.
6. **Projection** — the existing pure `apply(state, event)` folds each stamped
   event over the committed state to produce the candidate state (incremental —
   never a re-fold of the whole log). `apply` stamps `state_version` (D4).
7. **Validation** — `validate(candidate_state)`; blocking violations →
   `StateValidationError`, candidate discarded (D5); warnings retained for the
   result.
8. **Commit / Updated State** — State Updater atomically installs the candidate
   triple; Version Manager now reports `v+k`; `AppendResult` returned. Readers
   holding old snapshots are unaffected (D1) and re-read at their leisure.

```mermaid
sequenceDiagram
    autonumber
    participant C as Client (agent/manager)
    participant F as Facade (Engagement)
    participant G as Concurrency Guard
    participant A as Sequence Allocator
    participant P as Projection (apply)
    participant V as Validation (runner)
    participant U as State Updater

    C->>F: get_state() + current_version() = v
    Note over C: decide; build event(s) (seq=0)
    C->>F: append_events(events, expected_version=v)
    F->>G: admit?(events, v)
    alt admission or version check fails
        G-->>F: reject(reason)
        F-->>C: EventAdmissionError / VersionConflictError
    else admitted
        G-->>F: ok
        F->>A: allocate(v+1..v+k)
        A-->>F: stamped copies
        F->>P: apply* (committed state → candidate state)
        P-->>F: candidate state (state_version = v+k)
        F->>V: validate(candidate)
        alt blocking violations
            V-->>F: report(valid=false)
            F-->>C: StateValidationError(report)  — nothing committed
        else valid (± warnings)
            V-->>F: report(valid=true)
            F->>U: commit(candidate log, state, v+k)
            U-->>F: committed
            F-->>C: AppendResult(version=v+k, warnings)
        end
    end
```

# 6. Failure Scenarios (architectural behavior, not implementation)

| Scenario | Behavior |
|---|---|
| **Version conflict** | Detected by the Guard before allocation. Nothing allocated, nothing applied, engagement unchanged. `VersionConflictError(expected, actual)` gives the caller everything a retry policy needs. Recovery is always: fresh read → re-derive events → new append. |
| **Duplicate sequence** | **Impossible at the write path by construction** (single allocator, atomic commit). It can therefore only appear in an *externally supplied* log (M1.8 load, M1.9 replay, `from_state`+log scenarios) — where the D2 boundary integrity check catches it and refuses to fold (`ReplayIntegrityError`). Defense in depth, at the only place the defect can enter. |
| **Invalid event** | Wrong `engagement_id`, pre-assigned `seq`, duplicate `event_id`, or a Pydantic-invalid payload (unconstructible, so it cannot even be submitted). Rejected at admission; no version consumed; engagement unchanged. |
| **Projection failure** | A reducer raises on a stamped event (would indicate a reducer defect — reducers are total over the 49-event union and completeness-tested). The candidate is discarded, the error propagates as-is (a bug must be loud, not swallowed), the committed state is untouched because `apply` is pure and worked on it read-only. No seq gap results: allocation was only ever candidate-local. |
| **Validation failure** | The candidate post-state has blocking violations. Candidate discarded, `StateValidationError(report)` raised with the full report, engagement unchanged. By D5's post-state rule, a currently-invalid state accepts only corrective events. |
| **Partial append** | Cannot exist. The batch is judged once, committed once; the commit is a single atomic replacement. Observers see version `v` or `v+k`, never anything between. |
| **Crash during append** | M1.7 state is in-memory: a process crash loses the engagement regardless of *when* it happens — there is no partially-appended durable artifact because there is no durable artifact. What M1.7 contributes is the **invariant M1.8 must preserve**: commit is a single atomic point, and durability ordering (write log entry → fsync → advance version/snapshot) must keep it that way. That ordering is specified and tested in M1.8; designing it is explicitly out of scope here. |

# 7. Performance

**Complexity goals** (the contract; benchmarks verify the constants):

| Operation | Goal | Why it holds |
|---|---|---|
| Single append | **O(|state|), independent of log length** | Incremental `apply` (shallow model copies) + one validation pass; never a re-fold of the log. |
| Batch append (k events) | **O(k · apply) + one validation** | One guard, one commit, one judgment (D5). |
| Conflict detection | **O(1)** | Integer compare (+ O(1) `event_id` set lookup). |
| Version lookup | **O(1)** | Version Manager holds the integer. |
| `get_state()` | **O(|state|)** | Deep copy (D1) — the deliberate price of soundness. |

**Numeric expectations**, anchored to measured baselines (Apple M3, single-run
methodology of docs/performance): validation measured ~71µs @ 100 objects,
~301µs @ 1k, ~4.4ms @ 10k; reducer applications are single-digit µs (projection
folds 10k events in ~33ms). Expected: **single append well under 1ms** at
engagement scale (≤1k objects, validation-dominated); batch append of 100 events
≈ 100 applies + 1 validation — same order as a single append at the same state
size; `get_state()` snapshot cost expected in the tens–hundreds of µs (to be
measured — it is a new baseline, not a guess to be asserted).

Policy unchanged: these are **regression baselines, not targets**; no optimization
without a measurement that justifies it (per the projection-baseline precedent).
All numbers land in `docs/performance/` (append + validation baselines, TD-010).

# 8. Testing Strategy (no implementation)

- **Unit tests** — one file per component contract: Guard (admission matrix ×
  version compare), Allocator (contiguity, stamping copies, caller instance
  untouched), Version Manager (invariant triangle: `current_version` ==
  `state_version` == log length), State Updater (commit atomicity), facade
