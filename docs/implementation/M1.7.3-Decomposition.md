# M1.7.3 Decomposition — Append Pipeline

M1.7.3 (append pipeline + concurrency guard + facade event API) is the
highest-risk component of M1.7: it introduces the system's only mutation path.
It is therefore decomposed into **eight independently testable slices**, ordered
so that every slice builds only on closed, tested predecessors, and the
dangerous parts (orchestration, atomicity) arrive only after their inputs
(contracts, allocation, guarding) are proven in isolation. No slice approaches
the 300-line implementation ceiling; the largest (S4) is budgeted ≈180 lines.

**Module layout** (mirrors the M1.6 `validation/` pattern — small modules, one
concern each, orchestration confined to one file):

```
packages/state/append/
  __init__.py      # curated internal exports
  errors.py        # S1 — AppendError, VersionConflictError, EventAdmissionError
  result.py        # S1 — AppendResult (frozen)
  versioning.py    # S2 — VersionManager
  sequencing.py    # S2 — SequenceAllocator
  guard.py         # S3 — ConcurrencyGuard (pure decisions)
  commit.py        # S4 — StateUpdater (the single commit point)
  pipeline.py      # S4 — AppendPipeline (the only orchestrator)
tests/state/append/  # one test module per slice
```

`state/append/` stays **internal** until S5 wires the facade; only S5 touches
the public surface (the extension M1.3 pre-authorized).

## Dependency graph

```
S1 (errors + AppendResult)
 └─▶ S2 (VersionManager + SequenceAllocator)
      └─▶ S3 (ConcurrencyGuard)
           └─▶ S4 (AppendPipeline + StateUpdater)   [uses M1.7.2 apply, M1.6 validate]
                └─▶ S5 (Facade event API, single append + validate()→report)
                     └─▶ S6 (Multi-event append semantics)
                          ├─▶ S7 (Property + stress tests)
                          └─▶ S8 (Append-path benchmarks)
```

Strictly linear through S6; S7 and S8 are independent of each other.

---

## S1 — Error hierarchy + AppendResult (contract primitives)

- **Objective:** the append API's result and error contracts exist exactly as
  design §3 specifies, before anything can raise or return them.
- **Scope:** `AppendError(StratAgentError)`; `VersionConflictError(expected,
  actual)`; `EventAdmissionError(reason, event_id)`; `AppendResult` (frozen:
  `version`, `first_seq`, `last_seq`, `appended`, `warnings: list[Violation]`);
  `append/__init__.py` exports.
- **Out of scope:** any logic that raises/produces them; `ReplayIntegrityError`
  (M1.7.4's slice, per design).
- **Files:** new `packages/state/append/{__init__,errors,result}.py`; new
  `tests/state/append/test_contracts.py`; `CHANGELOG.md`.
- **Dependencies:** none (uses existing `StratAgentError`, `Violation`).
- **Acceptance criteria:** error attributes and human-readable messages match
  design §3; `AppendResult` is frozen and carries all five fields; inheritance
  chain correct (`isinstance(x, AppendError)` and `StratAgentError`).
- **Automated tests:** construction + attribute + message-content tests per
  error; frozen-mutation rejection on `AppendResult`; inheritance assertions.
- **Risks:** minimal — pure declarations. Only risk is contract drift from
  design §3, mitigated by writing tests against the design text.
- **Estimated complexity:** Low (≈80 impl lines).

## S2 — VersionManager + SequenceAllocator

- **Objective:** the two stateful-arithmetic components, proven in isolation.
- **Scope:** `VersionManager` (owns the committed version integer;
  `current_version`, `current_sequence == current_version + 1`,
  `advance_to(n)` guarded strictly monotonic). `SequenceAllocator.stamp(events,
  first_seq)` → stamped **frozen copies** with contiguous seqs; caller
  instances byte-identical after stamping.
- **Out of scope:** any decision-making (guarding), any commit, any facade.
- **Files:** new `packages/state/append/{versioning,sequencing}.py`; new
  `tests/state/append/test_versioning_sequencing.py`; `CHANGELOG.md`.
- **Dependencies:** S1 (monotonicity violation raises `AppendError`).
- **Acceptance criteria:** contiguity `first_seq … first_seq+k-1`; stamped
  events are new objects (originals untouched — frozen events copied via
  `model_copy`); `current_sequence == current_version + 1` always;
  `advance_to` rejects non-forward moves.
- **Automated tests:** stamp of 1/k events (contiguity, identity, original
  unmutated incl. metadata); version arithmetic; monotonic guard (equal,
  backward, skip-forward cases per contract).
- **Risks:** frozen-model copying subtleties (nested `EventMetadata` copy) —
  covered by identity tests reusing the M1.7.1 walker technique.
- **Estimated complexity:** Low–Medium (≈90 impl lines).

## S3 — ConcurrencyGuard

- **Objective:** every admission/conflict decision of design D3/D6, as pure
  functions that never mutate anything.
- **Scope:** admission checks (engagement_id match; `seq == 0` unassigned
  sentinel; unknown `event_id` vs a caller-supplied id-set; non-empty batch)
  raising `EventAdmissionError`; version compare (`expected == current` passes;
  `<` and `>` raise `VersionConflictError(expected, actual)`).
- **Out of scope:** orchestration (who calls the guard, in what order); the
  R/W-matrix role check (M1.7.6 seam, data-only per D6).
- **Files:** new `packages/state/append/guard.py`; new
  `tests/state/append/test_guard.py`; `CHANGELOG.md`.
- **Dependencies:** S1 (errors). (S2 not required — the guard compares
  integers it is handed.)
- **Acceptance criteria:** the full decision matrix from design §6 is
  implemented and each cell tested; guard functions are pure (no state, no
  mutation of inputs).
- **Automated tests:** admission matrix — wrong engagement, pre-assigned seq,
  duplicate event_id, empty batch, all-clear; version matrix — equal/stale/
  ahead; error payloads carry the reason and both versions.
- **Risks:** subtle contract drift (e.g., forgetting `expected > current` is
  also a conflict) — the test matrix is written cell-by-cell from the design
  table to prevent it.
- **Estimated complexity:** Low–Medium (≈80 impl lines).

## S4 — AppendPipeline + StateUpdater (orchestration core)

- **Objective:** the guarded, atomic, all-or-nothing append — the heart of
  M1.7 — over a committed `(log, state, version, id-set)` quadruple.
- **Scope:** `StateUpdater` (single commit point: atomic replacement of the
  committed references); `AppendPipeline.append(events, expected_version) ->
  AppendResult` running admission → guard → allocate → incremental `apply` →
  `validate` (blocking violations ⇒ `StateValidationError`; warnings into the
  result) → commit. Candidate discarded wholesale on any failure; no seq
  consumed by failed attempts.
- **Out of scope:** the facade (S5); batch-specific semantics beyond what the
  general pipeline naturally does (asserted in S6); property/stress suites
  (S7).
- **Files:** new `packages/state/append/{commit,pipeline}.py`; new
  `tests/state/append/test_pipeline.py`; `CHANGELOG.md`.
- **Dependencies:** S1–S3; M1.7.2 `apply`; M1.6 `validate`.
- **Acceptance criteria:** design §5 workflow step-for-step; design §6 failure
  table row-for-row — on every failure path the committed quadruple is
  **byte-identical** to pre-attempt (snapshot compare) and the version is
  unconsumed; on success `result.version == expected + k`, log grew by the
  stamped copies, state reflects the fold, state_version stamped (M1.7.2).
- **Automated tests:** happy path (single event, pipeline-level); each failure
  row: admission reject / version conflict / validation reject (invalid
  post-state) — each asserting no-commit + correct error; warning pass-through
  into `AppendResult`; idempotency (same event_id twice); incremental-apply
  equivalence for a short canonical log (`project(log) == committed state`).
- **Risks:** the highest of any slice — atomicity bugs (partial commit) and
  candidate/committed aliasing. Mitigations: prepare-commit is a reference
  swap of objects built from pure functions; the byte-identical-on-failure
  assertions run on **every** failure test; S1–S3 arrive pre-proven so this
  slice contains orchestration only.
- **Estimated complexity:** **Medium–High (≈180 impl lines)** — the risk
  ceiling of the decomposition, deliberately reached with every dependency
  already tested.

## S5 — Facade event API (single append) + `validate()` report

- **Objective:** the public surface of design §3 — the M1.3-reserved event-API
  extension — plus the D5 `validate() -> ValidationReport` upgrade.
- **Scope:** `Engagement` gains internal `(log, pipeline)` wiring;
  `append_event(event, *, expected_version) -> AppendResult`;
  `current_version()`; `current_sequence()`; `validate()` now returns the
  M1.6 `ValidationReport`; `EngagementProtocol` extended in lockstep; public
  exports (`AppendResult`, the three error types) added to `state.__all__`;
  facade-frozen-API and allowlist tests updated (documented extension).
- **Out of scope:** `append_events` (S6); doc-file updates beyond CHANGELOG
  (API reference lands with S6 so the event API is documented once, complete).
- **Files:** `packages/state/facade.py`, `packages/state/__init__.py`;
  `tests/state/test_facade.py`, `tests/state/append/test_facade_append.py`;
  `CHANGELOG.md`.
- **Dependencies:** S4.
- **Acceptance criteria:** design §3 contracts verbatim (result fields, error
  types, caller's event never mutated, engagement unchanged on failure);
  invariant triangle `current_version() == get_state().metadata.state_version
  == committed log length` after every append; `validate()` returns a report
  (`valid` bare state ⇒ `report.valid`); M1.7.1 snapshot semantics unaffected.
- **Automated tests:** facade-level append happy/conflict/admission/validation
  paths; triangle invariant walk over several appends; `validate()` report
  round-trip; updated frozen-surface test (exactly the six M1.3 methods + the
  four M1.7 additions); allowlist test.
- **Risks:** public-surface churn — contained by updating the two surface
  tests in the same commit; `validate()` return-type change is the documented
  D5 refinement (no external consumers).
- **Estimated complexity:** Medium (≈120 impl lines).

## S6 — Multi-event append (batch semantics)

- **Objective:** `append_events` — one guard, contiguous allocation, **one**
  final validation, all-or-nothing — as approved in design §3.
- **Scope:** `append_events(events, *, expected_version)` on facade +
  protocol; empty-batch rejection (`EventAdmissionError`); batch rows of the
  failure table (mid-batch invalid event ⇒ nothing committed);
  `docs/api/EngagementState.md` + `docs/api/Events.md` updated with the
  complete event API + seq contract (D2 text).
- **Out of scope:** property/stress suites (S7); benchmarks (S8).
- **Files:** `packages/state/facade.py`, `packages/state/append/pipeline.py`
  (only if the batch path needs a distinct entry — expected ≈0–20 lines);
  `tests/state/append/test_batch.py`; both API docs; `CHANGELOG.md`.
- **Dependencies:** S5.
- **Acceptance criteria:** `first_seq == expected+1`, `last_seq == expected+k`;
  intermediate states not judged, final state judged once; batch conflict and
  mid-batch failure leave the engagement byte-identical; empty batch rejected;
  docs document the full API.
- **Automated tests:** batch happy path (k=3, seq bookkeeping, single
  validation observed via a state valid only at batch end — e.g., evidence +
  finding arriving in one batch); batch atomicity failure cases; empty batch;
  `AppendResult` bookkeeping.
- **Risks:** "validate once at the end" subtlety — pinned by the
  valid-only-at-end test.
- **Estimated complexity:** Low–Medium (≈60 impl lines).

## S7 — Property + stress tests

- **Objective:** the design §8 randomized guarantees over the finished
  pipeline — tests only, no implementation lines.
- **Scope:** seeded pseudo-random generators (no new dependency); properties:
  (1) **fold equivalence** — for generated admissible append sequences
  *beginning with `EngagementCreated`* (the canonical birth event — see note),
  `project(committed_log) == committed state`; (2) snapshot isolation under
  random snapshot mutation; (3) contiguity 1..N after hundreds of interleaved
  appends; (4) stale-writer stress — two simulated writers, occasional
  deliberately-stale expected_version, zero lost updates, byte-identical
  rejection semantics; (5) cross-engagement determinism (same appends ⇒
  identical states incl. `state_version`).
- **Note (recorded honestly):** fold equivalence is stated over logs whose
  first event is `EngagementCreated` because `project()` folds from the empty
  placeholder state while `Engagement.create()` builds real metadata directly;
  the `EngagementCreated` reducer deterministically rebuilds metadata from the
  event, after which the two histories converge. The generator therefore
  always opens with `EngagementCreated` — matching the real lifecycle (ADR-002:
  every engagement's first event is `EngagementCreated`).
- **Files:** new `tests/state/append/test_properties.py`; `CHANGELOG.md`.
- **Dependencies:** S6. **Out of scope:** benchmarks.
- **Acceptance criteria:** all five properties hold over ≥ hundreds of seeded
  iterations; failures reproduce from the printed seed.
- **Automated tests:** the slice *is* tests.
- **Risks:** flakiness from unseeded randomness — every generator takes an
  explicit seed; runtime kept < ~2s so the gate stays fast.
- **Estimated complexity:** Medium (0 impl lines; ≈200 test lines).

## S8 — Append-path benchmarks

- **Objective:** baseline numbers for the new write path (design §7), in-suite.
- **Scope:** `tests/perf/test_append_bench.py`: single append at 4 state
  scales; batch 10/100/1k; conflict-rejection path. (The `get_state` snapshot
  benchmark and the consolidated `docs/performance` write-up remain in
  M1.7.7 as approved — this slice covers the pipeline the moment it exists so
  regressions are caught early.)
- **Files:** new `tests/perf/test_append_bench.py`; `CHANGELOG.md`.
- **Dependencies:** S6 (independent of S7).
- **Acceptance criteria:** benchmarks run in the automated suite with the
  established single-run methodology; results order-of-magnitude consistent
  with design §7 expectations (append ≪ 1 ms at engagement scale) — recorded,
  not asserted.
- **Automated tests:** the benchmarks themselves.
- **Risks:** none material; numbers are baselines, not gates.
- **Estimated complexity:** Low (≈50 test lines).
