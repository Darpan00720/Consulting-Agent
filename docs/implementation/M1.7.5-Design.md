# M1.7.5 Design — Remaining Validation Rules + Traceability Disposition

## 1. Objective

Close the two validation debts the Architecture Checkpoint scheduled into
M1.7 [Verified: TD register rows TD-002, TD-004; M1.7-Design D6 table]:

- **TD-002:** the four missing lifecycle **gate-entry precondition rules**
  (LIFE-005…LIFE-008) from ADR-002 §Validation Rules → "Required fields
  (gate preconditions)".
- **TD-004:** a **traceability disposition table** so that *every* ADR-002
  validation rule maps exactly once to its enforcement point — registry rule,
  record-level constructor rule (M1.1), boundary mechanism (M1.7.2/M1.7.3/
  M1.7.4), or an explicit deferral with its owning milestone.

Nothing else changes: no event model, projection, append pipeline, replay
integrity, or persistence design is touched [constraint, restated].

## 2. Scope

- Four new rules appended to the **existing** lifecycle registry
  (`state/validation/lifecycle.py`) — no new validation group; pure,
  deterministic, IO-free, independent validators; orchestration stays in the
  runner untouched [Verified: M1.6 architecture].
- Frozen rule-id namespace grows additively: `LIFE-005`…`LIFE-008`
  [Verified: rule-id freeze discipline, traceability doc].
- `scripts/generate_traceability.py` extended to emit a **disposition
  section** (markdown + JSON) covering all 25 ADR-002 validation-rule line
  items; a completeness test guards it.
- Regenerated `docs/implementation/traceability-ADR-002.md` +
  `traceability.json`.

## 3. Out of Scope

- Approval-actor rules (gate approver roles, no-self-approval) — **M6**, per
  the recorded TD-003 deferral [Verified: checkpoint + M1.7-Design D6; the
  data they need, `QualityGate.by`, is already captured].
- Post-completion mutation & event immutability enforcement — write-boundary
  concerns (§6 dispositions), not snapshot rules; the pipeline is frozen.
- Concurrency rules — implemented (M1.7.3) or M6-owned (§6).
- Any change to `runner.py`, existing rules, groups, severities, or the
  public API (`state.__all__` untouched; S6 freeze test must pass unmodified).

## 4. Remaining ADR-002 validation rules

Source text [Verified — ADR-002 §Validation Rules, "Required fields (gate
preconditions)"]: six preconditions are listed; two are already enforced
(Generate-Report → LIFE-001, Complete → LIFE-002 [Verified: M1.6 registry +
tests]). The remaining four, exactly as M1.7-Design D6 scheduled them:

| Rule id | Description (state-level formulation) | Rationale | Severity | Evidence | Class |
|---|---|---|---|---|---|
| **LIFE-005** | If `status` is `planning` or any later forward phase: `classification` is present (its `primary_archetype` is a required field, so presence suffices [Verified: scoping.py:69]), `problem.real_question` is set [Verified: optional field, scoping.py:36], and every gap with `criticality == LOAD_BEARING` has `status ∈ {ANSWERED, ASSUMED}` [Verified enums: GapCriticality.LOAD_BEARING; GapStatus] | ADR-002: "Enter Planning: `case_type.primary_archetype` + `real_question` present; all `load_bearing` gaps are `answered` or `assumed`" | ERROR | ADR-002 §Validation Rules line 492 | [Verified] |
| **LIFE-006** | If `status` is `analysis` or later: `issue_tree` is non-empty and `plan` is not None. (The "every leaf has an owner" half is **already** STRUCT-001 [Verified: M1.6] — cross-referenced, not duplicated) | ADR-002: "Enter Specialist Analysis: non-empty `issue_tree` (every leaf has an `owner`) + an `engagement_plan`" | ERROR | ADR-002 line 493 | [Verified] |
| **LIFE-007** | If `status` is `review` or later: every issue-tree **leaf** has `status == ANSWERED` [Verified enum: IssueNodeStatus.ANSWERED; leaf = node no other node references as `parent`, the STRUCT-001 definition]. (The "every finding has ≥1 evidence_ref" half is **already** STRUCT-002 — cross-referenced) | ADR-002: "Enter Reviewer: every issue-tree leaf `status=answered`; every finding has ≥1 `evidence_ref`" | ERROR | ADR-002 line 494 | [Verified] |
| **LIFE-008** | If `status` is `challenge` or later: `reviewer_notes` is present with `verdict == APPROVED` [Verified: ReviewerNotes.verdict optional; ReviewVerdict.APPROVED] | ADR-002: "Enter Challenger: `review.verdict=approved`" | ERROR | ADR-002 line 495 | [Verified] |

**"At-or-beyond" semantics (design decision, flagged):** each rule fires when
the *current* `status` sits at or beyond its phase in the forward order
already encoded in `validation/lifecycle.py::_FORWARD` [Verified: module
constant] — because a precondition of entering phase X must still hold while
in any later phase. Terminal statuses (`FAILED`, `ABORTED`) are **exempt**: an
engagement may fail at any point, and requiring un-entered phases'
preconditions of a failed engagement would be wrong [Inference — ADR-002 does
not address preconditions of terminal states; consistent with LIFE-001's
existing status-scoped pattern]. Rework loops need no special casing: after a
`ReviewerRejected` → back-to-`analysis` transition, `status` is `analysis`, so
LIFE-008 correctly does not fire [Verified: legal transitions in `_ALLOWED`].

Nothing else remains [Verified — exhaustive walk of the 25 ADR-002 line items
in §6; every item maps to an existing mechanism, one of these four rules, or a
recorded deferral. No rules were invented beyond the ADR text].

## 5. Validation ownership

| Rule | Owner |
|---|---|
| LIFE-005…008 | `state.validation` lifecycle registry (this milestone) — pure `state -> list[Finding]` validators, registry-based, runner-orchestrated, no new group |
| Leaf-owner, finding-evidence halves | already STRUCT-001 / STRUCT-002 — unchanged |
| Everything else in ADR-002 | per the §6 disposition table — no responsibility moves across layers |

The append pipeline automatically enforces the new rules at write time with
zero changes [Verified: the pipeline validates every candidate post-state via
the runner, which aggregates registries — a new registry entry is picked up by
construction]. Consequence (flagged): appends that produce states violating
LIFE-005…008 — e.g. a `PhaseTransitioned(intake → classifying → … → planning)`
without a classification — will now be **rejected** by the pipeline
(`StateValidationError`). This is the rules doing their job; existing tests
are unaffected (verified reasoning: current tests transition at most to
`classifying`/`reporting`-invalid cases already expecting rejection — to be
re-confirmed by the gate).

## 6. Traceability disposition (TD-004)

Generator change: `scripts/generate_traceability.py` gains a static,
reviewed **disposition list** — data, not behavior — emitted as a second
markdown table and a `dispositions` array in `traceability.json`. Chain per
rule: **ADR rule → Validator → Test → traceability document**, exactly once.

For the four new rules the chain is fully mechanical (the existing generator
already scans registries and slug-named tests) [Verified: generator code]:

```
ADR-002 "Enter Planning"      → LIFE-005 → lifecycle._planning_preconditions   → test_life_005_* → matrix row
ADR-002 "Enter Analysis"      → LIFE-006 → lifecycle._analysis_preconditions   → test_life_006_* → matrix row
ADR-002 "Enter Reviewer"      → LIFE-007 → lifecycle._review_preconditions     → test_life_007_* → matrix row
ADR-002 "Enter Challenger"    → LIFE-008 → lifecycle._challenge_preconditions  → test_life_008_* → matrix row
```

Disposition classes for the full 25-item ADR-002 walk (each item appears in
exactly one row; the table below summarizes — the generated document lists

**Completeness guard:** a new test asserts the generated disposition table
covers every ADR-002 §Validation-Rules line item exactly once and that every
"registry" row's rule id exists in `ALL_RULES` (so the matrix cannot drift).

## 7. Risks

- **Medium — new rules reject previously "valid" states.** Any state at
  `planning`+ without classification/real-question now carries an ERROR. No
  production states exist [Verified: pre-release]; existing tests build either
  early-phase or already-invalid fixtures. Residual risk: M1.5 reducer tests
  project mid-phase logs and assert fields, not validity — unaffected. The
  gate proves it.
- **Low — leaf-status semantics.** LIFE-007 uses the STRUCT-001 leaf
  definition (nodes nothing references as parent). A single-node tree's node
  is a leaf; consistent, documented.
- **Low — disposition table staleness.** Static data could rot; the
  completeness test pins it to the ADR line items and the live registry.
- **Flagged decision for approval:** the post-completion-mutation disposition
  row assigns enforcement to the append-admission policy at **M1.8** (when
  the pipeline next opens for persistence wiring) with M6's manager as the
  process-level backstop. Alternative: accept it as permanently
  boundary-deferred. Either way it is recorded, not silently dropped.

## 8. Definition of Done

1. LIFE-005…008 in `state/validation/lifecycle.py`: pure, deterministic,
   IO-free, independent, registry-based, runner-orchestrated; **no new
   groups; no runner change; no public-API change** (S6 freeze test passes
   unmodified; `git diff` clean on pipeline/integrity/projection/events).
2. Tests: slug-named negative + positive/branch tests per rule (the M1.6
   pattern); terminal-status exemption tested; the at-or-beyond semantics
   tested at a later phase (violation still detected at `analysis` for a
   planning precondition).
3. Traceability: generator emits the disposition section (md + JSON);
   documents regenerated; completeness test green; 25/25 ADR items mapped
   exactly once.
4. Coverage: `lifecycle.py` and the generator's new code 100%; full six-step
   gate green; zero regressions; frozen files untouched.
5. CHANGELOG entry (TD-002 and TD-004 closed); BACKLOG updated if it lists
   them; DoD report with the flagged-decision disposition confirmed.
