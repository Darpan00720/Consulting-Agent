# M1.8 Design — Persistence (append / save / load)

# 1. Objective

Give an engagement **durable, file-backed storage**: persist its event log and a
state snapshot, and load them back into a working (append-capable) `Engagement`
— restored through the frozen M1.7 seams (`verify_pair` → `project` →
`AppendPipeline`), preserving every v1.0 invariant. Round-trip:
`create → append* → save → load → append*` yields an engagement identical to the
one saved [design goal, per M1-Decomposition §M1.8].

# 2. Scope

- A **file store** for one engagement: an append-only event log + a state
  snapshot under `engagements/<slug>/`.
- **save** (atomic), **append** (durable, append-only), **load** (parse →
  verify → reconstruct an append-capable `Engagement`).
- The store is the **single IO-touching component** in the state domain
  (see DD-1); it composes the frozen seams and adds no new state semantics.

# 3. Out of Scope

- **Rebuild / recovery / replay** — M1.9 [Verified: M1-Decomposition §M1.9 owns
  "rebuild / projection-driven / snapshots / recovery"]. M1.8 loads a
  **consistent** persisted pair; a crash-torn pair (log ahead of snapshot) is an
  M1.9 concern (see §9, DD-6).
- **Event schema upcasting** (TD-007) — the format *records* `schema_version`
  and M1.8 *rejects* unknown versions with a clear error; actual upcasting is
  deferred [Verified: M1.7.4 §R16; BACKLOG TD-007 → M1.8/M1.9].
- **Distributed / multi-process / concurrent writers**, DB backends (Ruflo
  AgentDB is M7), cross-engagement indexes.
- Any change to projection, validation, the append pipeline, events, or the
  facade's existing behaviour.

# 5. Component Decomposition (proposed; slice-gated like M1.7.3)

| Component | Responsibility |
|---|---|
| **Paths** | resolve `engagements/<slug>/{events.log,state.json,manifest.json}`; slug validation |
| **LogWriter** | append event line(s) as NDJSON; fsync; append-only |
| **LogReader** | read NDJSON → `tuple[Event, ...]` (TypeAdapter) |
| **SnapshotWriter** | atomic write of `state.json` (+ manifest) via temp+rename+fsync |
| **SnapshotReader** | read `state.json` → `EngagementState` |
| **EngagementStore** | the facade of persistence: `save(engagement)`, `append(...)`, `load(slug) -> Engagement`; composes the above + `verify_pair` + `AppendPipeline` |

Delivered as small approval-gated slices (S1 paths+format, S2 write, S3 read+
verify, S4 store round-trip, S5 docs/tests) — mirroring M1.7.3's discipline.

# 7. Complete Persistence Lifecycle

```mermaid
sequenceDiagram
    participant C as Caller
    participant S as EngagementStore
    participant FS as Filesystem
    participant I as verify_* (M1.7.4)
    participant P as project + AppendPipeline

    Note over C,FS: SAVE
    C->>S: save(engagement)
    S->>S: log = <committed log>, snapshot = state
    S->>FS: append new events → events.log ; fsync
    S->>FS: write state.json(+manifest) via temp → fsync → rename

    Note over C,FS: LOAD
    C->>S: load(slug)
    S->>FS: read events.log → log ; read state.json → snapshot
    S->>I: verify_log(log) ; verify_pair(log, snapshot)
    I-->>S: ok (else ReplayIntegrityError)
    S->>P: AppendPipeline(snapshot, log=log, append_supported=True)
    S-->>C: Engagement (append-capable)
```

Note [Inference]: the store reads the committed log from the pipeline. The facade
does **not** expose the log (`to_json` is state-only) [Verified], so `save` needs
read access to `Committed.log` — a decision (DD-3): either the store operates on
the pipeline/`Committed` (internal) or a small additive accessor is introduced.

# 8. Atomicity Guarantees

- **Snapshot (`state.json`, `manifest.json`):** write to `*.tmp`, `fsync`,
  `os.replace` (atomic rename on POSIX) [Inference — standard atomic-write
  idiom; **I don't know from the available evidence** of any existing atomic-write
  helper in the repo → new code].
- **Log (`events.log`):** open in append mode, write the new event line(s),
  `fsync`. Append of a single write is the durable unit.
- **Ordering (preserves the M1.7 atomic-commit invariant):** persist the **log
  first (append+fsync), snapshot second (atomic rename)**. Rationale: the log is
  the source of truth (ADR-002); a snapshot is a derivable optimization. A crash
  between the two leaves log ⊇ snapshot, never the reverse — recoverable by
  re-projection (M1.9), never a lost event.
- **PER-invariant:** at rest, a *successfully completed* save yields a pair where
  `verify_pair` passes. Torn writes are detectable (§9), never silently loaded.

# 9. Failure & Recovery Model

| Failure | Behaviour |
|---|---|
| Crash mid-log-append | last line may be partial → `LogReader` detects a non-parseable trailing line and **rejects** (or truncates-to-last-valid — DD-5); no partial event is admitted |
| Crash between log-fsync and snapshot-rename | log ahead of snapshot → `verify_pair` raises `STATE_VERSION_MISMATCH` (**recoverable**) [Verified — integrity R11/R13]. **M1.8 surfaces it; the fix (re-project) is M1.9** (DD-6) |
| Missing snapshot, log present | `verify_pair` cannot run; M1.8 **refuses** with a clear error pointing to M1.9 rebuild (DD-6) |
| Corrupt/garbled JSON | parse error → `PersistenceError` (new) with path + line; never a silent load |
| `schema_version` unknown/future | rejected (TD-007 deferral); clear error, no upcasting |
| Foreign/mixed engagement in a log | `verify_log` R7/R18 raises `ENGAGEMENT_MISMATCH` [Verified] |

M1.8's contract: **it loads consistent pairs and refuses inconsistent ones with a
precise, typed error.** No repair, no fabrication (the S5 no-fabrication rule at
rest). Repair = M1.9.

# 10. File Formats

- **`events.log`** — NDJSON, one `Event` JSON object per line, in `seq` order,
  append-only. Round-trip [Verified]. UTF-8, `\n`-terminated.
- **`state.json`** — `EngagementState.model_dump_json()` (identical to the
  facade's `to_json()` output) [Verified].
- **`manifest.json`** (proposed, DD-2) — `{engagement_id, slug, tenant_id,
  event_count, last_seq, state_version, projection_version, schema_version,
  written_at}` — an O(1) integrity pre-check and an explicit record of what the
  pair claims (redundant with, and cross-checked against, the log+snapshot).

# 11. Versioning Strategy

- **`projection_version`** [Verified — M1.7.2/M1.7.4]: the snapshot carries it;
  `verify_pair` treats `> current` as fatal (`PROJECTION_FUTURE`) and `< current`
  as recoverable (`PROJECTION_STALE` → re-project = M1.9). M1.8 loads only
  matching versions; stale/future are surfaced, not silently migrated.
- **`schema_version`** (events) [Verified — EventMetadata default 1]: recorded
  per event; M1.8 rejects unknown values (TD-007 defers upcasting).
- **Store format version:** the `manifest` carries a `format_version` so the
  on-disk layout can itself evolve (additive, additive-frozen namespace).

# 12. Snapshot Strategy

M1.8 persists a **single latest snapshot** (`state.json`) alongside the full log
— the snapshot is an optimization so load need not re-fold from empty. Periodic /
historical snapshots and snapshot-driven fast replay are **M1.9** [Verified:
M1-Decomposition §M1.9]. On load, the snapshot is *verified against* the log
(`verify_pair`), never trusted blindly.

# 13. Replay Strategy

M1.8 does **not** replay. Load reconstructs the pipeline directly from the
persisted `(log, snapshot)` pair after `verify_pair`. Full projection-driven
**replay/rebuild** (fold from the log when the snapshot is absent/stale/corrupt)
is M1.9. The seam is ready: `project(log)` + `AppendPipeline(state, log=log)`.

# 14. Validation Strategy

- **On load:** `verify_log` (R1–R18) then `verify_pair` (state_version/identity/
  projection) — structural integrity of the persisted artifact [Verified].
- **Business validity:** the loaded state is what was previously committed, so it
  already passed the append-time validation gate; M1.8 does **not** re-run the
  M1.6 runner on load (it may be an invalid-but-persisted state only if it was
  never appended — impossible via the pipeline). `Engagement.validate()` remains
  available to callers.

# 15. Complexity Analysis

| Operation | Time | Space |
|---|---|---|
| append k events | **O(k)** write + one fsync | O(k) |
| save snapshot | **O(\|state\|)** serialize + atomic rename | O(\|state\|) |
| load | **O(n)** read + **O(n)** verify_log + **O(n·apply)** project (build snapshot only if rebuilding — else O(1) trust-after-verify) + O(\|state\|) snapshot parse | O(n + \|state\|) |
| verify on load | O(n) (integrity single pass) [Verified — M1.7.4 §7] | O(n) |

Baselines to be measured (M1.8 bench, extending `docs/performance/baselines.md`):
save/load at 10/100/1k/10k events — same single-run methodology.

# 16. Stable Invariants (PER-xx)

| ID | Invariant | Enforced by |
|---|---|---|
| PER-1 | Round-trip fidelity: `load(save(e))` yields an engagement with identical committed log, state, and version | round-trip tests |
| PER-2 | Log is append-only NDJSON in contiguous `seq` order | LogWriter (append mode); verify_log R1–R5 |
| PER-3 | A completed save produces a `verify_pair`-passing pair | save ordering + verify on load |
| PER-4 | Load never admits a torn/inconsistent artifact — it raises a typed error | verify_log/verify_pair + PersistenceError |
| PER-5 | Snapshot writes are atomic (temp→fsync→rename); no partial `state.json` observable | atomic-write idiom |
| PER-6 | Durability ordering: log fsync precedes snapshot rename (no lost event on crash) | save sequence |
| PER-7 | Loaded engagements are append-capable (`append_supported=True`); replayed history + new appends share one contiguous seq space | AppendPipeline(log=…, append_supported=True) |
| PER-8 | No fabrication/repair on load (no renumber, synthesize, or silent migrate) | store contract |
| PER-9 | Store is the only IO in `state`; projection/validation/pipeline remain IO-free | module boundary + review |
| PER-10 | Unknown `projection_version`/`schema_version` are refused with a clear error | verify_pair + schema check |

# 19. Risks

- **[Medium] IO boundary erosion (DD-1):** putting file IO in `state` breaks the
  Verified IO-free invariant. Mitigated by isolating all IO in one clearly-marked
  store module + PER-9 boundary test; or by placing the store outside the core
  (DD-1 option b).
- **[Medium] engagements/<slug> collision:** the `/solve-case` plugin already
  writes `01-intake.md`…`report.md` there [Verified — engagements/README.md].
  M1.8's `events.log`/`state.json` coexist; slugs must not collide, and
  persisted data probably should be gitignored (currently not) [Verified: not
  ignored]. Decision DD-4.
- **[Medium] M1.8/M1.9 boundary (DD-6):** "load" vs "rebuild". Torn/stale pairs
  need re-projection = M1.9; M1.8 must draw the line cleanly or it absorbs M1.9
  scope.
- **[Low] fsync/atomicity portability:** POSIX `os.replace` atomicity assumed;
  Windows semantics differ. Target is macOS/Linux dev [Verified: .python-version,
  Darwin]; note it.
- **[Low] partial-line detection:** trailing torn NDJSON line — reject vs
  truncate-to-valid (DD-5).

# 20. Architectural Concerns

The one genuine architectural question is **DD-1** — introducing IO into a
domain that ADR-001/Architecture-v1.0 keep strictly IO-free. Everything else
composes existing frozen seams. If DD-1 keeps IO in a single, boundary-tested
module (or a sibling package) and the facade stays frozen (DD-3b), M1.8 adds
persistence **without** touching the frozen v1.0 architecture — which is the
stated goal.

# Decisions Requiring Approval

- **DD-1 — Placement / IO boundary.** (a) `packages/state/store.py` (single
  IO module inside `state`, boundary-tested) vs (b) a sibling package
  `packages/persistence/` importing `state` (keeps `state` IO-free). **Recommend
  (b)** — it preserves the Verified IO-free invariant cleanly. *This is the
  headline decision.*
- **DD-2 — manifest.json:** include a manifest (recommended, for O(1) integrity +
  explicit version record) or log+snapshot only.
- **DD-3 — Load/save API surface:** (a) add `save`/`load` to `Engagement`/
  `EngagementProtocol` (breaks the frozen 10-method surface → explicit approval +
  freeze-test update) vs (b) a **separate `EngagementStore`** object
  (`save(engagement)`, `load(slug) -> Engagement`) — additive, facade frozen.
  **Recommend (b)**; requires a small read accessor for `Committed.log` (the
  store works on the pipeline/`Committed`, which is internal — no public change).
- **DD-4 — engagements/ hygiene:** gitignore `engagements/*/` (keep README) and
  confirm coexistence with the plugin's markdown artifacts under the same slug.
- **DD-5 — torn trailing log line on load:** reject with `PersistenceError`
  (strict) vs truncate-to-last-valid. **Recommend reject** (M1.8 loads consistent
  artifacts; recovery is M1.9).
- **DD-6 — M1.8/M1.9 boundary:** M1.8 = **load consistent pairs, refuse
  inconsistent ones** (no re-projection); all rebuild/recovery (stale/missing/
  torn snapshot) = M1.9. **Recommend this split** (matches M1-Decomposition).
- **DD-7 — save trigger:** explicit `store.save(engagement)` / `store.append(...)`
  (durable-on-demand) for M1.8; auto-persist-on-append is a later concern.
  **Recommend explicit.**
