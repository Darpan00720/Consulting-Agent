## 1. Objective

Provide the **authoritative reconstruction path**: given a persisted (or
otherwise supplied) event log, rebuild a verified, append-capable `Engagement`
whose state is the canonical projection of that log — and provide the
**recovery** path that upgrades a stale persisted snapshot by re-projection.
Replay is the milestone that finally makes an adopted-from-storage engagement
**writable again** (removes the `append_supported=False` limitation for
log-backed engagements) [Verified — Architecture-v1.0 Extension Points].

## 2. Scope (replay owns)

1. **Rebuild authoritative state from an event log** — fold a verified log to a
   canonical `EngagementState` and produce an append-capable `Engagement`.
2. **Rebuild stale snapshots** — the `PROJECTION_STALE` recovery path deferred
   from M1.8 (DD-6) [Verified].
3. **Projection upgrades** — re-project when a persisted snapshot's
   `projection_version < PROJECTION_VERSION` [Verified — `verify_pair` message
   prescribes "discard the snapshot and re-project the verified log"].
4. **Deterministic replay** — replay is a pure function of the log.
5. **Replay verification** — reuse `verify_log`/`verify_pair` as the at-rest gate.
6. **Replay recovery workflow** — the orchestration that turns a stale/degraded
   persisted artifact into a current, verified, append-capable engagement.

## 3. Out of scope (frozen — reused, never modified)

- **Persistence / save / load** — `EngagementStore` (M1.8) is frozen. Replay may
  *call* `load`/`save`, but adds/changes nothing in `packages/persistence`.
- **Append** — `AppendPipeline`, `make_committed`, the phase order, the version
  triangle (M1.7.3) are frozen.
- **Validation** — M1.6 runner/rules frozen.
- **Replay integrity** — `verify_log`/`verify_pair`/`ReplayErrorCode` (M1.7.4)
  frozen; reused verbatim.
- **Ownership** — `state.ownership` (M1.7.6) frozen; unenforced until M6.
- **Projection engine** — `project`/`apply`/`PROJECTION_VERSION` frozen; replay
  never re-implements the fold.
- **Architecture v1.0** — ADR-001…005; the 10-method facade surface; event
  immutability. Any change here requires a **new ADR** [Verified].

## 5. Replay workflow (the mandatory contract)

For a log with **no trusted snapshot** (rebuild from log alone):

```
verify_log(log)                 # at-rest gate; fatal on any defect
        ↓
state = project(log)            # canonical fold; pv == PROJECTION_VERSION
        ↓
verify_pair(log, state)         # re-runs verify_log; confirms the canonical pair
        ↓
AppendPipeline(state, log=log,  # constructs genesis CandidateCommit and calls
               append_supported=True)   #   make_committed internally (P23)
        ↓
Engagement(pipeline)            # append-capable, seq continues at current_version(log)
```

- **`make_committed` is invoked transitively** by `AppendPipeline` construction
  [Verified — `pipeline.py:74-81`]; replay does **not** call it separately (that
  would double-construct) [Inference]. See Decision **D-C**.
- Because `state = project(log)` always yields `projection_version ==
  PROJECTION_VERSION`, `verify_pair` always passes for a log that passed
  `verify_log` [Verified — mirrors the M1.8 canonical-pair guarantee]. Replay of
  a *verified* log therefore cannot fail at the fold or pair step.
- For a log **with** a trusted persisted snapshot, this is exactly what
  `EngagementStore.load` already does [Verified]. Decision **D-C/D-D** covers
  whether replay wraps `load` or operates on an already-loaded log.

## 6. Recovery workflow (stale snapshot → current)

Trigger: a persisted `(log, snapshot)` where `verify_pair` raises
`PROJECTION_STALE` (`snapshot.projection_version < PROJECTION_VERSION`) — the
one **recoverable** code [Verified].

```
load bytes → decode → verify_log(log)          # log itself must be valid (fatal otherwise)
        ↓  verify_pair raises PROJECTION_STALE  (recoverable)
discard stale snapshot
        ↓
state' = project(log)                           # re-project under current impl → pv = PROJECTION_VERSION
        ↓
verify_pair(log, state')                        # now passes
        ↓
AppendPipeline(state', log=log, append_supported=True) → Engagement
        ↓  (optional, Decision D-D)
EngagementStore.save(engagement)                # persist upgraded (log, project(log)); manifest rewritten
```

Note: in M1.8, `projection_stale` on load is **unreachable by construction**
(persistence always stores `project(log)`) [Verified — M1.8 report]. It becomes
reachable only when `PROJECTION_VERSION` is bumped in a **future** milestone and
an **older** artifact is then loaded. M1.9 designs the path so it is ready when
that happens; whether M1.9 also performs the write-back is Decision **D-D**.

## 7. Projection upgrade workflow

- **Mechanism** [Verified from `verify_pair` prescription]: upgrade = *discard
  the stale snapshot and re-project the verified log*; the log is the source of
  truth, the snapshot is a derived cache. Re-projection yields the current
  `PROJECTION_VERSION` by construction.
- **`PROJECTION_FUTURE` is not upgradable** — a snapshot from a *newer*
  implementation is **fatal** (cannot down-project); it requires a newer binary,
  not recovery [Verified — fatal code].
- **Migrations / upcasting of events** (changing event *schemas*, not the
  projection) were "deferred to M1.8 by approval" and not implemented
  [Verified — `integrity.py:27`]. Whether M1.9 needs event upcasting, or only
  whole-log re-projection, is Decision **D-E**. Current evidence: re-projection
  alone covers `projection_version` drift; event *schema* evolution is a
  separate, unaddressed concern.

## 8. Failure taxonomy

**Existing codes are reused unchanged** (`ReplayErrorCode` is additive-frozen;
M1.9 must not modify it) [Verified].

| Condition | Existing code | Class | Replay disposition |
|---|---|---|---|
| Log structurally invalid | `SEQUENCE_*`, `GENESIS_*`, `ENGAGEMENT_*`, `EVENT_ID_DUPLICATE`, `UNASSIGNED_EVENT` | **Fatal** | replay refuses; propagate |
| Snapshot from future impl | `PROJECTION_FUTURE` | **Fatal** | refuse; needs newer binary |
| Snapshot/log identity or version disagreement | `ENGAGEMENT_MISMATCH`, `STATE_VERSION_MISMATCH` | **Fatal** | refuse; corrupt/foreign artifact |
| Stale snapshot | `PROJECTION_STALE` | **Recoverable** | re-project (§7) |
| Torn/missing/corrupt artifact at read | `TornWriteError`/`MissingArtifactError`/`CorruptArtifactError` (persistence) | **Fatal** | surfaced by `EngagementStore`; replay does not repair |

**New replay errors — where required (identification only, no implementation):**

- **Integrity level:** *none required* [Inference] — a verified log cannot fail
  the fold (`project` never fails) or the pair (canonical pv), so replay adds no
  new integrity codes.
- **Orchestration level:** *possibly one* — a thin wrapper for "recovery
  attempted on a fatal (non-recoverable) artifact" or "recovery requested but the
  log is itself invalid." This would be **additive**, in a **new** module, and
  must not touch `ReplayErrorCode`. Whether it is warranted vs. simply
  re-raising the existing fatal error is Decision **D-G**.

Classification of the recovery outcomes: **Fatal** (refuse, no recovery) /
**Recoverable** (`PROJECTION_STALE` → re-project) / **Deferred** (event-schema
upcasting, cross-version migration — out of scope; a future milestone, D-E).

## 9. Complexity

| Step | Complexity | Basis |
|---|---|---|
| `verify_log` | O(n) single pass | [Verified — M1.7.4 §7] |
| `project` (fold) | ~O(n), mild superlinear observed | [Verified — baselines: `project`@10k ≈ 54 ms] |
| `verify_pair` | O(n) | [Verified — baselines] |
| `AppendPipeline` construction | O(n) (`current_version` + genesis candidate) | [Inference from `commit.py`/`pipeline.py`] |
| **Full replay(log)** | **O(n)** dominated by the fold | [Inference] |
| Recovery (re-project) | O(n) — one extra fold vs. a trusting load | [Inference] |

No step is worse than the projection fold already baselined; replay adds
constant orchestration, not new asymptotic cost [Inference].

## 11. Test strategy (property/stress; absorbs deferred S7)

Design only — no tests written this phase.

- **Determinism (RP-003):** for random valid logs, two independent `replay`
  calls produce identical `Committed` and byte-identical snapshots.
- **Fold equivalence (RP-004):** `replay(log).state == project(log)` across
  generated logs; and replayed state equals a persisted snapshot that passes
  `verify_pair`.
- **Repeated-replay stability (RP-005):** replaying a replay's own log is a
  fixpoint.
- **Corruption rejection (RP-006/007):** mutate a valid log (drop/reorder/renumber
  an event; forge a future/stale/foreign snapshot) → assert the exact existing
  `ReplayErrorCode`, fatal vs. recoverable per taxonomy.
- **Projection-upgrade recovery (RP-008/009):** construct a `PROJECTION_STALE`
  artifact (snapshot pv < current) → recovery yields pv == `PROJECTION_VERSION`
  and a `verify_pair`-passing pair; append works afterward.
- **Large-log replay (stress):** replay at 10 / 100 / 1k / 10k (and 100k subject
  to Decision **D-F**) events; assert success + version triangle, never timing.
- **Frozen-boundary (RP-014):** `git diff` zero on `packages/state` +
  `packages/persistence`; source-scan that replay imports the seams and
  re-implements none.

Property tests imply a generator dependency (e.g. Hypothesis) not currently in
the dev group — adopting it is Decision **D-F** (tooling).

## 12. Performance strategy

Follow the existing baseline methodology verbatim [Verified —
`docs/performance/baselines.md`]: `pytest-benchmark`, `benchmark.pedantic(rounds=1,
iterations=1)`, single cold run per scale, **no latency assertions** (regression
references only), fixtures built through the real pipeline.

- **Scales:** 10 / 100 / 1,000 / 10,000 / 100,000 events (100k pending **D-F**).
- **Measure:** replay throughput (`replay(log)` end to end), projection
  throughput (`project(log)` in isolation — extends the existing `project`
  baseline to 100k), replay-verification throughput (`verify_log` +
  `verify_pair`).
- **Record:** a persistence-style table appended to `docs/performance/baselines.md`.
- **Caveat:** `get_state` deep-copy is O(|state|) and ~73 ms at 10k already
  [Verified]; a 100k full-state materialization may be heavy — hence D-F.

## 13. Risks

- **[Low] `projection_stale` path is exercised only synthetically** until a real
  `PROJECTION_VERSION` bump. Mitigation: tests construct stale artifacts directly.
- **[Medium] Public-surface creep.** Exposing replay could tempt adding a facade
  method, which changes the frozen 10-method surface and needs an ADR. Mitigation:
  Decision **D-B** decides location/surface *before* coding.
- **[Medium] Recovery/write-back blurs the persistence boundary.** Auto-saving an
  upgraded snapshot risks implicit persistence (DD-7 forbids autosave).
  Mitigation: Decision **D-D**.
- **[Low] 100k benchmark cost/memory.** Mitigation: D-F caps or defers.
- **[Low] Property-test tooling.** New dependency (Hypothesis) needs approval (D-F).
