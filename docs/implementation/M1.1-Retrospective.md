# Retrospective — M1.1 (Evidence & Assumption Ledgers)

**Date:** 2026-06-30

## What worked well
- The approval-gated flow (plan → implement → DoD report) kept scope tight: M1.1
  touched only `packages/state/ledgers.py` and its tests.
- The **modular per-type validation registry** satisfied the maintainability
  directive cleanly and is already earning its keep — adding an evidence type is a
  one-line registration, no validator surgery.
- The deterministic gate (ruff / black / mypy-strict / pytest) plus coverage gave
  objective, real evidence of "done" — no claims without output.
- 100% coverage on new code; 26 tests; zero regressions to M0.

## What should be improved
- The M0/M1 boundary needed an explicit note (schema **not** regenerated in M1.1).
  Future sub-milestone plans should state schema impact up front.
- A generated coverage artifact (`.coverage`) appeared untracked; tool side-effects
  should be git-ignored **when the tool is introduced**, not as an afterthought.

## Risks discovered
- **Breadth risk for M1.2:** ~20 section models + several enums raise the chance of
  omitting an ADR-002 field → mitigate with a section-coverage test traced 1:1 to
  ADR-002.
- **Python version drift:** system Python is bleeding-edge 3.14 while the project is
  pinned to 3.12; pinning is holding, but watch dependency wheels on each `uv sync`.

## Technical debt introduced
- **TD-001** — `Evidence.source` overloaded naming; evaluate `reference` / `origin`
  before external APIs stabilize (tracked in `BACKLOG.md`; review during M1).
- Minor: `Assumption.owner` typed `str` pending the agent enum (M3) — expected, not
  a concern.

## Actions before M1.2
1. ✅ Record TD-001 in the backlog; start `CHANGELOG.md`; file this retrospective.
2. ✅ Git-ignore coverage artifacts (done as an M1.1 follow-up commit).
3. The M1.2 plan must state **schema impact** and **section optionality/default
   semantics** explicitly.
4. Await approval of the M1.2 plan before writing any M1.2 code.
